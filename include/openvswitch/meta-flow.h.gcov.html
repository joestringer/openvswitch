      34                 :            : /* Open vSwitch fields
      35                 :            :  * ===================
      36                 :            :  *
      37                 :            :  * A "field" is a property of a packet.  Most familiarly, "data fields" are
      38                 :            :  * fields that can be extracted from a packet.
      39                 :            :  *
      40                 :            :  * Some data fields are always present as a consequence of the basic networking
      41                 :            :  * technology in use.  Ethernet is the assumed base technology for current
      42                 :            :  * versions of OpenFlow and Open vSwitch, so Ethernet header fields are always
      43                 :            :  * available.
      44                 :            :  *
      45                 :            :  * Other data fields are not always present.  A packet contains ARP fields, for
      46                 :            :  * example, only when its Ethernet header indicates the Ethertype for ARP,
      47                 :            :  * 0x0806.  We say that a field is "applicable" when it is it present in a
      48                 :            :  * packet, and "inapplicable" when it is not, and refer to the conditions that
      49                 :            :  * determine whether a field is applicable as "prerequisites".  Some
      50                 :            :  * VLAN-related fields are a special case: these fields are always applicable,
      51                 :            :  * but have a designated value or bit that indicates whether a VLAN header is
      52                 :            :  * present, with the remaining values or bits indicating the VLAN header's
      53                 :            :  * content (if it is present).  See MFF_VLAN_TCI for an example.
      54                 :            :  *
      55                 :            :  * Conceptually, an inapplicable field does not have a value, not even a
      56                 :            :  * nominal ``value'' such as all-zero-bits.  In many circumstances, OpenFlow
      57                 :            :  * and Open vSwitch allow references only to applicable fields.  For example,
      58                 :            :  * one may match a given field only if the match includes the field's
      59                 :            :  * prerequisite, e.g. matching an ARP field is only allowed if one also matches
      60                 :            :  * on Ethertype 0x0806.
      61                 :            :  *
      62                 :            :  * (Practically, however, OVS represents a field's value as some fixed member
      63                 :            :  * in its "struct flow", so accessing that member will obtain some value.  Some
      64                 :            :  * members are used for more than one purpose, e.g. the "tp_src" member
      65                 :            :  * represents the TCP, UDP, and SCTP source port, so the value read may not
      66                 :            :  * even make sense.  For this reason, it is important to know whether a field's
      67                 :            :  * prerequisites are satisfied before attempting to read it.)
      68                 :            :  *
      69                 :            :  * Sometimes a packet may contain multiple instances of a header.  For example,
      70                 :            :  * a packet may contain multiple VLAN or MPLS headers, and tunnels can cause
      71                 :            :  * any data field to recur.  OpenFlow and Open vSwitch do not address these
      72                 :            :  * cases uniformly.  For VLAN and MPLS headers, only the outermost header is
      73                 :            :  * accessible, so that inner headers may be accessed only by ``popping''
      74                 :            :  * (removing) the outer header.  (Open vSwitch supports only a single VLAN
      75                 :            :  * header in any case.)  For tunnels, e.g. GRE or VXLAN, the outer header and
      76                 :            :  * inner headers are treated as different data fields.
      77                 :            :  *
      78                 :            :  * OpenFlow and Open vSwitch support some fields other than data fields.
      79                 :            :  * "Metadata fields" relate to the origin or treatment of a packet, but they
      80                 :            :  * are not extracted from the packet data itself.  One example is the physical
      81                 :            :  * port on which a packet arrived at the switch.  "Register fields" act like
      82                 :            :  * variables: they give an OpenFlow switch space for temporary storage while
      83                 :            :  * processing a packet.  Existing metadata and register fields have no
      84                 :            :  * prerequisites.
      85                 :            :  *
      86                 :            :  * A field's value consists of an integral number of bytes.  Most data fields
      87                 :            :  * are copied directly from protocol headers, e.g. at layer 2, MFF_ETH_SRC is
      88                 :            :  * copied from the Ethernet source address and MFF_ETH_DST from the destination
      89                 :            :  * address.  Other data fields are copied from a packet with padding, usually
      90                 :            :  * with zeros and in the most significant positions (see e.g. MFF_MPLS_LABEL)
      91                 :            :  * but not always (see e.g. MFF_IP_DSCP).  A final category of data fields is
      92                 :            :  * transformed in other ways as they are copied from the packets, to make them
      93                 :            :  * more useful for matching, e.g. MFF_IP_FRAG describes whether a packet is a
      94                 :            :  * fragment but it is not copied directly from the IP header.
      95                 :            :  *
      96                 :            :  *
      97                 :            :  * Field specifications
      98                 :            :  * ====================
      99                 :            :  *
     100                 :            :  * Each of the enumeration values below represents a field.  The comments
     101                 :            :  * preceding each enum must be in a stylized form that is parsed at compile
     102                 :            :  * time by the extract-ofp-fields program.  The comment itself consists of a
     103                 :            :  * series of paragraphs separate by blank lines.  The paragraphs consist of:
     104                 :            :  *
     105                 :            :  *     - The first paragraph gives the user-visible name of the field as a
     106                 :            :  *       quoted string.  This is the name used for parsing and formatting the
     107                 :            :  *       field.
     108                 :            :  *
     109                 :            :  *       For historical reasons, some fields have an additional name that is
     110                 :            :  *       accepted as an alternative in parsing.  This name, when there is one,
     111                 :            :  *       is given as a quoted string in parentheses along with "aka".  For
     112                 :            :  *       example:
     113                 :            :  *
     114                 :            :  *           "tun_id" (aka "tunnel_id").
     115                 :            :  *
     116                 :            :  *       New fields should have only one name.
     117                 :            :  *
     118                 :            :  *     - Any number of paragraphs of free text that describe the field.  This
     119                 :            :  *       is meant for human readers, so extract-ofp-fields ignores it.
     120                 :            :  *
     121                 :            :  *     - A final paragraph that consists of a series of key-value pairs, one
     122                 :            :  *       per line, in the form "key: value." where the period at the end of the
     123                 :            :  *       line is a mandatory part of the syntax.
     124                 :            :  *
     125                 :            :  * Every field must specify the following key-value pairs:
     126                 :            :  *
     127                 :            :  *   Type:
     128                 :            :  *
     129                 :            :  *     The format and size of the field's value.  Some possible values are
     130                 :            :  *     generic:
     131                 :            :  *
     132                 :            :  *         u8: A one-byte field.
     133                 :            :  *         be16: A two-byte field.
     134                 :            :  *         be32: A four-byte field.
     135                 :            :  *         be64: An eight-byte field.
     136                 :            :  *
     137                 :            :  *     The remaining values imply more about the value's semantics, though OVS
     138                 :            :  *     does not currently take advantage of this additional information:
     139                 :            :  *
     140                 :            :  *         MAC: A six-byte field whose value is an Ethernet address.
     141                 :            :  *         IPv6: A 16-byte field whose value is an IPv6 address.
     142                 :            :  *         tunnelMD: A variable length field, up to 124 bytes, that carries
     143                 :            :  *                   tunnel metadata.
     144                 :            :  *
     145                 :            :  *   Maskable:
     146                 :            :  *
     147                 :            :  *     Either "bitwise", if OVS supports matching any subset of bits in the
     148                 :            :  *     field, or "no", if OVS only supports matching or wildcarding the entire
     149                 :            :  *     field.
     150                 :            :  *
     151                 :            :  *   Formatting:
     152                 :            :  *
     153                 :            :  *     Explains how a field's value is formatted and parsed for human
     154                 :            :  *     consumption.  Some of the options are fairly generally useful:
     155                 :            :  *
     156                 :            :  *       decimal: Formats the value as a decimal number.  On parsing, accepts
     157                 :            :  *         decimal (with no prefix), hexadecimal with 0x prefix, or octal
     158                 :            :  *         with 0 prefix.
     159                 :            :  *
     160                 :            :  *       hexadecimal: Same as decimal except nonzero values are formatted in
     161                 :            :  *         hex with 0x prefix.  The default for parsing is *not* hexadecimal:
     162                 :            :  *         only with a 0x prefix is the input in hexadecimal.
     163                 :            :  *
     164                 :            :  *       Ethernet: Formats and accepts the common format xx:xx:xx:xx:xx:xx.
     165                 :            :  *         6-byte fields only.
     166                 :            :  *
     167                 :            :  *       IPv4: Formats and accepts the common format w.x.y.z.  4-byte fields
     168                 :            :  *         only.
     169                 :            :  *
     170                 :            :  *       IPv6: Formats and accepts the common IPv6 formats.  16-byte fields
     171                 :            :  *         only.
     172                 :            :  *
     173                 :            :  *       OpenFlow 1.0 port: Accepts an OpenFlow well-known port name
     174                 :            :  *         (e.g. "IN_PORT") in uppercase or lowercase, or a 16-bit port
     175                 :            :  *         number in decimal.  Formats ports using their well-known names in
     176                 :            :  *         uppercase, or in decimal otherwise.  2-byte fields only.
     177                 :            :  *
     178                 :            :  *       OpenFlow 1.1+ port: Same syntax as for OpenFlow 1.0 ports but for
     179                 :            :  *         4-byte OpenFlow 1.1+ port number fields.
     180                 :            :  *
     181                 :            :  *     Others are very specific to particular fields:
     182                 :            :  *
     183                 :            :  *       frag: One of the strings "no", "first", "later", "yes", "not_later"
     184                 :            :  *         describing which IPv4/v6 fragments are matched.
     185                 :            :  *
     186                 :            :  *       tunnel flags: Any number of the strings "df", "csum", "key", or
     187                 :            :  *         "oam" separated by "|".
     188                 :            :  *
     189                 :            :  *       TCP flags: See the description of tcp_flags in ovs-ofctl(8).
     190                 :            :  *
     191                 :            :  *   Prerequisites:
     192                 :            :  *
     193                 :            :  *     The field's prerequisites.  The values should be straightfoward.
     194                 :            :  *
     195                 :            :  *   Access:
     196                 :            :  *
     197                 :            :  *     Either "read-only", for a field that cannot be changed via OpenFlow, or
     198                 :            :  *     "read/write" for a modifiable field.
     199                 :            :  *
     200                 :            :  *   NXM:
     201                 :            :  *

     204                 :            :  *     parentheses, followed by "since v<x>.<y>" specifying the version of Open
     205                 :            :  *     vSwitch that first supported this field in NXM (e.g. "since v1.1" if it
     206                 :            :  *     was introduced in Open vSwitch 1.1).
     207                 :            :  *
     208                 :            :  *     The NXM name must begin with NXM_OF_ or NXM_NX_.  This allows OVS to
     209                 :            :  *     determine the correct NXM class.
     210                 :            :  *
     211                 :            :  *     If the field does not have an NXM field assignment, specify "none".
     212                 :            :  *
     213                 :            :  *   OXM:
     214                 :            :  *
     215                 :            :  *     If the field has an OXM field assignment, then this specifies the OXM
     216                 :            :  *     name of the field (e.g. "OXM_OF_ETH_SRC"), followed by its nxm_type in
     217                 :            :  *     parentheses, followed by "since OF<a>.<b> v<x>.<y>" specifying the
     218                 :            :  *     versions of OpenFlow and Open vSwitch that first supported this field in
     219                 :            :  *     OXM (e.g. "since OF1.3 and v1.10" if it was introduced in OpenFlow 1.3
     220                 :            :  *     and first supported by Open vSwitch in version 1.10).
     221                 :            :  *
     222                 :            :  *     Some fields have more than one OXM field assignment.  For example,
     223                 :            :  *     actset_output has an experimenter OXM assignment in OpenFlow 1.3 and a
     224                 :            :  *     standard OXM assignment in OpenFlow 1.5.  In such a case, specify both,
     225                 :            :  *     separated by commas.
     226                 :            :  *
     227                 :            :  *     OVS uses the start of the OXM field name to determine the correct OXM
     228                 :            :  *     class.  To support a new OXM class, edit the mapping table in
     229                 :            :  *     build-aux/extract-ofp-fields.
     230                 :            :  *
     231                 :            :  *     If the field does not have an OXM field assignment, specify "none".
     232                 :            :  *
     233                 :            :  * The following key-value pairs are optional.  Open vSwitch already supports
     234                 :            :  * all the fields to which they apply, so new fields should probably not
     235                 :            :  * include these pairs:
     236                 :            :  *
     237                 :            :  *   OF1.0:
